# Design: EnterpriseRAG v1.4 — Agent-First Architecture Inversion

Mode: Builder

## Problem Statement

EnterpriseRAG v1.3 architecture is "RAG-first, agent-as-mode": four parallel pipelines (`IngestionPipeline`, `QueryPipeline`, `AgentQueryPipeline`, `SwarmQueryPipeline`), where the agent variant is one of several query modes. The user wants to invert this: agent becomes the architectural core, RAG becomes one of several tools the agent calls. This is an architectural identity change, not a feature add.

Concretely: the v1.3 `AgentQueryPipeline` runs a single LLM with up to 5 tool-use iterations, but RAG retrieval is still hardcoded as the implicit "main" path. The desired v1.4 model is: requests enter an agent runtime → planner decomposes → executor fans out to tools (one of which is RAG) → synthesizer composes the answer.

## What Makes This Cool

The whoa for the target audience (peer RAG/AI engineers reading the open-source repo) lives in three things:

1. **Visible planner output as an SSE event stream.** Most agentic-RAG code paths are an opaque LLM tool-use loop. Surfacing `planner.plan` events (the decomposition the agent just made), `tool.span` events (each tool call boxed with timing/inputs/outputs), and `executor.parallel` markers (showing fan-out factor) makes the agent's reasoning legible — the same way a SQL query planner is.
2. **Parallel tool fan-out you can watch happen.** When the planner emits 4 sub-queries, the SSE timeline shows them executing simultaneously rather than serialized — the latency win and the architectural choice are visible in the same artifact.
3. **Enterprise constraints intact.** Multi-tenancy/RLS, JWT, audit log, 70%+ coverage gate are all still enforced under the new architecture. Most agent demos drop these. This one keeps them. That's the differentiator vs. LangGraph tutorials and toy AutoGPT clones.

## Constraints

- v1.3 multi-tenant model (PostgreSQL RLS, `tenant_id` thread-through) cannot break.
- JWT denylist + 32-char minimum + per-route rate limits (v1.0 hardening) preserved.
- Combined coverage floor 70% (v1.3 gate) applied to new modules from day one, not deferred.
- `MODEL_DIR` env var, structured logging, Tenacity retry on external calls — all the v1.0 OPS-* requirements remain.
- pgvector + HNSW backend stays; new architecture wraps it, doesn't replace it.
- No prototype code (Pydantic V2, mypy --strict, ruff) standard.
- Audience is open-source readers, so ABI-style stability of the new public surface (`/agent/v1/run` or equivalent) and example code completeness matter.

## Premises

1. The existing `AgentQueryPipeline` (Anthropic Tool Use, max 5 iterations, RAG embedded) can be refactored in place into planner+executor. No greenfield agent core needed.
2. RAG-as-tool means wrapping `QueryPipeline.run()` as a `RetrieveTool`; the existing hybrid retrieval + RRF fusion + reranker logic stays inside that tool, unchanged.
3. The SSE event stream for planner trace (`planner.plan` / `tool.span` / `executor.parallel`) is the wedge's whoa-bearing payload. It must land in v1.4 — not deferred to a polish milestone.
4. The 10x roadmap (Memory → Multi-agent swarm → Tool registry/MCP → Code-acting) all builds on top of the planner+executor+tool-registry foundation. Foundation first, expansions after, in that order.
5. Architectural inversion does not break any v1.3 invariant: multi-tenancy, RLS, JWT, audit log, 70% coverage gate. New design operates within these.

The user agreed with all five premises without revision.

## Approaches Considered

### Approach A: Incremental refactor of AgentQueryPipeline (chosen)

Refactor the existing `services/pipeline.py::AgentQueryPipeline` into three explicit collaborators:
- `Planner` — first LLM call returns a `ToolPlan` (list of `ToolCall` objects with declared parallelism).
- `Executor` — runs `ToolPlan` via `asyncio.gather`, yields `tool.span` events.
- `Synthesizer` — final LLM call composes the response from accumulated tool outputs.

`QueryPipeline.run()` is wrapped as `RetrieveTool` (hybrid retrieval + RRF + rerank stays internal). New skeletal tools: `WebSearchTool` (placeholder), `SQLTool` (placeholder), `MemoryTool` (deferred, reserves namespace for 10x #1).

SSE events flow through the existing `/query/stream` infrastructure; no new transport.

- Effort: M (one v1.4 milestone, 4-6 phases following GSD workflow)
- Risk: Low (in-place refactor of code we own, with existing test coverage as a backstop)
- Pros: Reversible per phase, no framework lock-in, planner trace already visible to demo audience, all v1.3 enterprise constraints preserved by construction.
- Cons: `Planner` schema is hand-rolled (no graph DSL); we own the trace event schema and must document it; less "industrial" looking than LangGraph code.
- Reuses: `AgentQueryPipeline`, `QueryPipeline`, `llm_client.call_agentic_turn` (provider-neutral tool-use loop), structured logger, SSE infra (`/query/stream`).

### Approach B: LangGraph foundation rewrite

Replace `AgentQueryPipeline` with a LangGraph `StateGraph`. Nodes: `planner`, `retrieve`, `sql`, `web`, `synthesizer`. State graph models the agent flow explicitly. LangGraph's native streaming output bridges into the SSE endpoint.

Existing `QueryPipeline` becomes the body of the `retrieve` node. `pyproject.toml` already has `langchain[eval]>=1.2.10` and `langchain-community>=0.3.14`, so LangGraph is the same dependency family.

- Effort: L (rewrite + test refresh; ~2 milestones realistically)
- Risk: Medium (LangGraph public API is still evolving; multi-tenant context has to be threaded explicitly through `state`; v1.3's 71.9% coverage on `pipeline.py` and dependent modules has to be re-earned).
- Pros: Industrial-grade graph DSL; existing community patterns for memory/persistence; native trace; future Memory feature has off-the-shelf approach.
- Cons: Framework lock-in; added stability surface (LangGraph version churn); breaks more existing tests; "wrote our own and inverted it" story is more credible to peers than "we adopted a framework."
- Reuses: `QueryPipeline` (as tool), `llm_client` (for the underlying LLM), pgvector layer.

### Approach C: Dual-track external agent shell

Leave the v1.3 RAG-first architecture untouched. Add a new top-level service `services/agent_core/` containing a self-built planner+executor runtime that calls the existing services (retriever, vectorizer, generator) as an internal SDK. New endpoint `/agent/v1/run` lives alongside `/query`. Agent runtime can be hot-swapped later (own implementation now, optionally LangGraph in v1.5).

- Effort: M-L
- Risk: Medium (two API surfaces to maintain; the "agent-first" reframing is delayed by one version because the old surface is still RAG-first).
- Pros: Zero impact on v1.3 users; allows side-by-side comparison of RAG-mode vs. agent-mode; gives optionality on the planner implementation.
- Cons: Doubles maintenance and docs surface; the architectural identity claim ("agent-first project") is half-true in v1.4.
- Reuses: All existing services as an internal SDK.

## Recommended Approach

**Approach A — incremental refactor.**

Rationale: the chosen audience (peer engineers reading the repo) will get their whoa from the *result* — planner trace and parallel fan-out visible in the SSE timeline — not from whether the implementation uses LangGraph. A delivers that result. It also has the lowest framework risk, preserves every v1.3 enterprise invariant by construction, and is the foundation all four 10x expansions (Memory, Swarm v2, MCP registry, Code-acting) explicitly require. B can still be revisited later if a specific 10x sub-feature (e.g. multi-agent debate) wants graph DSL — see Open Questions.

## Open Questions

1. **Planner output schema.** Pydantic model for `ToolPlan` — what fields? Tentative: `steps: list[ToolCall]`, `parallel_groups: list[list[int]]`, `rationale: str`. Decide in Phase 1 of v1.4 plan.
2. **Tool registration mechanism.** Static class registry (simple, MCP-deferred) vs. plugin discovery (more work, sets up MCP for 10x #3). Recommend static registry for v1.4 with a clean abstraction so MCP can replace it without callsite changes.
3. **Iteration cap policy.** v1.3's `max_iterations=5` was a hardcoded constant. New planner might want adaptive cap (per-plan) or per-tenant config. Recommend keep static for v1.4 to constrain blast radius.
4. **Backwards compatibility.** Keep `/query?agent_mode=true` working as a thin alias for `/agent/v1/run`? Or break it and document migration? Open-source repo means external users may exist — favor alias.
5. **Sub-agent reuse.** Is the v1.3 `SwarmQueryPipeline` a "tool" (one of the things `Executor` can dispatch) or a separate execution mode? Recommend tool: it composes naturally and the planner can choose it for queries that benefit from debate. Confirm in v1.4 Phase 1 discussion.
6. **Cross-model second opinion.** Skipped during this office-hours session. Worth running `codex` cold-read on this design before planning kicks off, to surface anything missed.

## Success Criteria

- v1.4 ships with `Planner`, `Executor`, `RetrieveTool`, and at least one additional tool (skeletal `WebSearchTool` or `SQLTool` is acceptable — proves the pluggability).
- `/query/stream` (or new `/agent/v1/run/stream`) emits at minimum these event types: `planner.plan`, `tool.span` (start, end, error), `executor.parallel` marker, `synthesizer.final`. Schemas are documented in `docs/`.
- A demo query that requires multi-hop reasoning (e.g. "compare X across regions A, B, C using docs and the metrics SQL table") produces a visible parallel fan-out in the SSE timeline. Captured as a recorded gif/asciinema in the README.
- v1.3 invariants intact: PostgreSQL RLS still enforces per-tenant isolation in tool calls, JWT/audit/rate-limit unchanged, combined coverage stays ≥70% (new modules included from day one).
- Latency: agentic queries with N parallel tools should be bounded by max(tool_latency), not sum. Logged and asserted in integration tests.
- README has an "Architecture" section that leads with agent-first; agentic-RAG framed as one tool the agent uses.
