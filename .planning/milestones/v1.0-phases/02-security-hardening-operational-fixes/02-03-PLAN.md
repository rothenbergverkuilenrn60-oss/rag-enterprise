In services/rules/rules_engine.py:

The challenge: Rule is currently a `@dataclass`. ABC + dataclass can coexist in Python via `ABCMeta` if done carefully. Per D-17, use plain class with `__init__` if ABC + dataclass causes issues.

Apply this approach (ABC + manual __init__, dropping @dataclass from Rule only):

1. ADD ABC imports at the top:
   ```python
   from abc import ABC, abstractmethod
   ```

2. REMOVE `@dataclass` decorator from Rule class only (keep @dataclass on RuleResult — it is not being changed).

3. CHANGE Rule class definition:
   ```python
   class Rule(ABC):
       """Abstract base class for all rules. Subclasses MUST implement check()."""

       def __init__(
           self,
           rule_id: str,
           description: str,
           stage: str,
           priority: int = 100,
           enabled: bool = True,
       ) -> None:
           self.rule_id = rule_id
           self.description = description
           self.stage = stage
           self.priority = priority
           self.enabled = enabled

       @abstractmethod
       def check(self, context: dict[str, Any]) -> RuleResult:
           """Execute this rule against the given context. Must return a RuleResult."""
           ...
   ```

4. All concrete subclasses (PromptInjectionRule, InputSanitizationRule, LLMOutputSafetyRule, SensitiveContentRule, OutOfScopeRule, AnswerQualityRule, FaithfulnessRule) already implement `check()`. They inherit from Rule via `@dataclass`. Since Rule is no longer a dataclass, these subclasses that are decorated with `@dataclass` will inherit from a plain ABC. This works in Python — `@dataclass` on a subclass of a non-dataclass ABC is fine.

   Verify no subclass uses `@dataclass` and relies on Rule's dataclass fields for `__init__` generation. If they do, each subclass's `@dataclass` decorator will generate its own `__init__` that calls super().__init__() only if written explicitly. Since the subclasses all pass field values via keyword arguments to `Rule(rule_id=..., description=..., stage=..., priority=...)`, they work as-is.

   Actually: the subclasses ARE decorated with `@dataclass` (they inherit from Rule which was a dataclass). After removing @dataclass from Rule, the subclasses still have @dataclass. The @dataclass decorator on a subclass that inherits from a non-dataclass class will only include the subclass's OWN fields. The inherited fields (rule_id, description, stage, priority, enabled) will NOT be part of the generated __init__ — they'll be set via super().__init__() only if the generated __init__ calls super().__init__().

   The safest approach per D-17: remove @dataclass from ALL Rule subclasses that have no additional fields (PromptInjectionRule, InputSanitizationRule, LLMOutputSafetyRule, SensitiveContentRule, OutOfScopeRule, AnswerQualityRule, FaithfulnessRule), since they define no dataclass fields of their own — they only define class-level constants (_INJECTION_PATTERNS etc.) and the check() method. Their instantiation uses keyword arguments that match Rule.__init__ parameters exactly.

   For each subclass decorated with `@dataclass` that has NO additional instance fields (only class-level re.compile constants and methods), remove the `@dataclass` decorator.
