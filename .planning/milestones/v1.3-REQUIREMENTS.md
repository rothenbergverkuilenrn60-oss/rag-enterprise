## v1 Requirements (final outcomes)

### Track E — Agentic Layer

#### REQ E-3 (AGENT-03): True fork-agent swarm with isolated sub-agent contexts ✅

**As a** user issuing a multi-dimension query (e.g., "审计上月所有未结案件的产假天数、病假规定、加班补偿政策") with `agent_mode=True`
**I want** the system to automatically decompose my query into N independent sub-queries and pursue each with a dedicated sub-agent (isolated message history, tool registry, and iteration budget)
**So that** complex multi-faceted questions are answered with full depth on each dimension, without one sub-question crowding out another's context

**Background:** v1.2 shipped `asyncio.gather` parallel tool-call burst within a single agent turn — multiple tool calls execute concurrently, but all share one message history and one iteration budget. True swarm is different: each sub-question gets its own agent with isolated context, its own tool-use loop, and its own stop condition.

**Outcome:** Validated 2026-05-09. `SwarmQueryPipeline` ships with `_decompose` (coordinator + N=1 fallback), `_run_sub_agent` (isolated `messages` per sub-query, no shared state), `asyncio.gather` parallel burst with `BaseException` isolation, all-failure synthesis short-circuit, `MAX_SWARM_AGENTS=5` + `MAX_SWARM_TURNS_PER_AGENT=5` caps, and audit log per swarm invocation. 7/7 acceptance criteria verified at [12-VERIFICATION.md](../phases/12-fork-agent-swarm/12-VERIFICATION.md).

**Acceptance criteria:**
- [x] `SwarmQueryPipeline` includes a coordinator that decomposes a multi-dimension query into N sub-queries.
- [x] Each sub-agent runs an independent `call_agentic_turn` loop with its own `messages: list`, `tools: list`, `max_iterations` budget; no shared state.
- [x] All N sub-agents execute concurrently via `asyncio.gather`; total swarm latency bounded by slowest sub-agent.
- [x] Coordinator collects N final answers and performs synthesis LLM call producing one unified response.
- [x] `MAX_SWARM_AGENTS` and `MAX_SWARM_TURNS_PER_AGENT` caps enforced.
- [x] Audit log per swarm invocation records N, per-agent turn count, per-agent tool calls, total latency, synthesis latency.
- [x] Unit tests cover decomposition, isolation, synthesis, caps. Integration test covers live multi-dimension query.

### Track NLU — Query Understanding

#### REQ NLU-02: LLM-based filter extractor fallback ✅

**As a** user whose query uses natural language to reference a page or section (e.g., "关于第三章的内容" instead of "第3章") that the regex extractor misses
**I want** the system to automatically fall back to an LLM call to extract the page/section filter
**So that** filtered retrieval works for the full range of Chinese query expressions, not just the patterns covered by the regex

**Outcome:** Validated 2026-05-09. `FilterExtractor` class composes regex-first then LLM-fallback. Cache layer reuses `utils/cache.py` (no hand-rolled MD5/Redis). `fallback_source` field on `ExtractionResult` traces which path produced the filter. 4/4 pipeline callsites migrated to `await get_filter_extractor().extract(query)`. 5/5 acceptance criteria verified at [13-VERIFICATION.md](../phases/13-llm-filter-fallback/13-VERIFICATION.md).

**Acceptance criteria:**
- [x] `FilterExtractor.extract` calls LLM only when regex returns empty.
- [x] LLM fallback result cached by query string; identical queries hit at most one LLM call per cache TTL.
- [x] Invalid LLM JSON / API errors caught silently → return empty filter, no exception propagation.
- [x] `fallback_source` field on returned filter (`"regex"` | `"llm"` | `None`) traces path.
- [x] Unit tests cover regex-hit (LLM never called), regex-miss → LLM-hit, JSON-fail, API-fail, cache-hit. Integration test for live LLM smoke.

### Track UI — Frontend

#### REQ UI-02: Frontend multi-file split and DOM modernization ✅

**As a** developer maintaining the EnterpriseRAG UI
**I want** the frontend JavaScript and CSS extracted from `static/ui.html` into separate `static/ui.js` and `static/ui.css` files, with the inline JS modernized to use current DOM APIs
**So that** the frontend is easier to read, edit, and test without opening a monolithic HTML file, and browser developer tools can apply source maps

**Outcome:** Validated 2026-05-09. `static/ui.html` references `ui.css` + `ui.js`; no inline `<script>` or `<style>` blocks; no inline `onclick`/`onsubmit`. JS IIFE-wrapped with `addEventListener` wiring; preserves try/catch + esc() escape logic + 6× `getElementById` lookups. main.py and StaticFiles symlink unchanged. AC#6 visual regression accepted on mechanical proxies (CSS byte-identical, HTML structural shell intact, all 4 served paths return HTTP 200) per user decision 2026-05-09 — browser smoke test recommended at first deploy. 6/6 must-haves verified at [14-VERIFICATION.md](../phases/14-frontend-split-and-dom-modernization/14-VERIFICATION.md).

**Acceptance criteria:**
- [x] `static/ui.html` contains no `<script>` blocks with inline JavaScript and no `<style>` blocks with inline CSS.
- [x] `static/ui.js` and `static/ui.css` exist; FastAPI `StaticFiles` serves them without configuration changes.
- [x] All `onclick=`, `onsubmit=`, and inline event handlers replaced with `addEventListener` calls.
- [x] DOM access uses `document.getElementById` / `document.querySelector` (6 occurrences preserved from inline form).
- [⊘] ES module imports + lightweight bundler — not adopted (criterion was optional; deferred until benefit emerges).
- [⚠️] Visual regression: accepted on mechanical proxies — live browser smoke test deferred to first deploy.

### Track TEST — Coverage

#### REQ TEST-04: Aggregate coverage across unit and integration pipelines ✅

**As a** developer reviewing CI results
**I want** the coverage report to reflect execution from both unit tests and integration tests combined
**So that** service paths exercised only by integration tests count toward the coverage floor, giving an accurate picture of actual test coverage

**Outcome:** Validated 2026-05-09. CI runs `unit-tests` (writes `.coverage.unit`) + `integration-tests` (writes `.coverage.integration` with `--cov-append`, `continue-on-error: true`); new `coverage-combine` job (`needs: [unit-tests, integration-tests]`, `if: always()`) downloads both artifacts, runs `coverage combine --keep`, then `coverage report --fail-under=70` (TEST-06 hard gate) and `diff-cover coverage.xml --compare-branch=v1.0 --fail-under=80` (TEST-03 migrated from unit-tests per Phase 15 D-05). Final `coverage-report` artifact contains `.coverage`, `.coverage.unit`, `.coverage.integration`, `coverage.xml`, `diff-cover.html`. 5/5 acceptance criteria verified at [15-UAT.md](../phases/15-coverage-combine-and-70-floor/15-UAT.md).

**Acceptance criteria:**
- [x] CI produces separate `.coverage.unit` and `.coverage.integration` files.
- [x] `coverage combine` produces `.coverage`; `coverage report` and `coverage xml` run on combined artifact.
- [x] Combined coverage report is source of truth for both global floor (TEST-06) and diff-cover gate (TEST-03).
- [x] CI artifact uploads combined coverage; `coverage-report` artifact bundles all data files + xml + diff-cover.html.
- [x] Per-file new-code diff-cover gate works on combined `.coverage` (no regression).

#### REQ TEST-06: Raise global coverage floor from 46% to 70% ✅

**As a** developer making changes to any service module
**I want** the CI to block merges that drop combined coverage below 70%
**So that** the codebase maintains a meaningful test baseline that prevents regression in core service paths

**Outcome:** Validated 2026-05-09. `pyproject.toml [tool.coverage.report]` sets `fail_under = 70`, `show_missing = true`, `precision = 1`. CI `coverage-combine` job runs `coverage report --fail-under=70` on combined artifact (threshold hard-coded inline; tampering pyproject.toml does not bypass — T-15-01-01 mitigation). Combined coverage 53.2% (pre-Wave-2) → 71.9% (post-Wave-2). 20 services/ modules below 70% at v1.2 close received new unit test files (one happy + one error path each). 5/5 acceptance criteria verified at [15-UAT.md](../phases/15-coverage-combine-and-70-floor/15-UAT.md).

**Acceptance criteria:**
- [x] `pyproject.toml [tool.coverage.report]` sets `fail_under = 70`.
- [x] CI step runs `coverage report --fail-under=70` on combined `.coverage` artifact.
- [x] Per-file diff-cover gate (TEST-03, `≥ 80%`) continues independently — not affected by floor change.
- [⚠️] All service modules below 70% at v1.2 close have new unit tests covering primary execution paths — **20/20 modules received new test files**; aggregate floor 71.9% met; 5 large modules (pipeline, llm_client, vector_store, retriever, extractor) still below per-module 70% individually but contribute to aggregate (deferred to v1.4 per CONTEXT D-04 scope boundary; documented in 15-02-SUMMARY.md).
- [x] Coverage report in CI artifacts shows per-module breakdown (`show_missing = true`).
