## Implementation Decisions

### README narrative thesis (SC1)

- **D-01:** README opening thesis: **"A Planner → Executor → Synthesizer agent. RAG is one of its tools."** Lead with the architectural inversion — this is an agent-first system, not a RAG-first system. SSE event stream appears under `## Observability` lower in the doc, not as the headline (it is the implementation mechanic, not the elevator pitch). Existing "RAG platform" framing is REPLACED.

- **D-02:** Section order in the rewritten README:
  1. Title + 2-line thesis ("A Planner → Executor → Synthesizer agent. RAG is one of its tools.")
  2. `## Quick demo` — asciinema cast embed + 1 paragraph explaining the parallel fan-out
  3. `## Architecture` — ASCII flow diagram: Request → Planner (emits ToolPlan) → Executor (parallel dispatch + as_completed) → Synthesizer → Response. One sentence per box.
  4. `## Tools the agent calls` — bullet list with one line each: `RetrieveTool` (hybrid pgvector + BM25 + RRF + reranker), `WebSearchTool` (placeholder), planned tools (`SQLTool`, `MCPTool` — Phase 19+/v1.5+).
  5. `## Platform features` — preserves the existing v1.3 README technical details (multi-tenant RLS, 6-stage ingest, 10-stage query pipeline, image extraction, async ingest, security, observability, streaming) but reframed as "what each tool / supporting service does."
  6. `## Observability` — SSE event stream paragraph + link to `docs/agent-architecture.md#event-schema-reference`.
  7. `## Quick start` — Docker compose + `make demo-agent` + minimal cURL example against `/agent/v1/run/stream`.
  8. `## Project status` — v1.4 release callout + link to milestones archive + CHANGELOG.

- **D-03:** Audience = peer engineers evaluating "is this worth using / forking / studying." Tone: confident, technical, no marketing language. Bullet density preferred over prose. Code snippets show real signatures, not pseudocode.

- **D-04:** Existing v1.3 README "Features" list is NOT preserved verbatim. It is reorganized and reframed under `## Platform features` with each item recast in agent-first terms. No information is lost; the framing is inverted.

### Demo query + recording format (SC3 + SC4)

- **D-05:** Demo query content: **"Across our compliance, finance, engineering, and HR knowledge bases, where do we mention 'data retention'?"** Plan = ONE parallel group of 4 `RetrieveTool` calls (one per KB shard), each fixture-mocked to ~0.5s tool latency. Same shape as Phase 18 SC4 latency assertion (4×0.5s parallel, total elapsed ∈ (450, 700) ms — bounded by max, not sum). Reuses the Phase 18 fixture pattern so the demo *is* the integration test.

- **D-06:** `make demo-agent` mechanics:
  - Stub LLM client (no API key required) — Planner returns a hardcoded `ToolPlan` with 4 parallel `RetrieveTool` calls; Synthesizer returns a hardcoded composed answer. Implementation detail: a tiny `services/agent/_demo_stubs.py` (or extension of an existing test stub) shared between the integration test and `make demo-agent`.
  - Stub tool registry — each `RetrieveTool` call returns a fixture `ToolResult` with 2-3 chunks per shard.
  - Spins up the Docker stack (`docker compose up -d`) per ROADMAP SC3, executes the demo, then prints the SSE event stream to stdout.
  - Deterministic, offline, CI-runnable. Trade-off: the answer text is fixture-shaped; the WHOA is the SSE event timeline (`planner.plan` → 4 × `tool.span.start` → 4 × `tool.span.end` → `executor.parallel(fan_out=4, group_latency_ms<700)` → `synthesizer.final`).
  - Exit code 0 on success. Non-zero exit on any event-shape mismatch.

- **D-07:** Recording format: **asciinema only (canonical)**. Cast file lives at `docs/demo.cast` (in-repo). README embeds via the standard GitHub-rendered `<a><img src="https://asciinema.org/a/...">` pattern OR a static gif rendered from the cast (`agg docs/demo.cast docs/demo.gif`) for fully-offline rendering — Phase 19 picks one. No separate gif workflow if asciinema renders inline. Recording is generated by `make demo-agent` itself (asciinema rec wrapping the demo invocation).

- **D-08:** No new asciinema dependency on contributors who run `make demo-agent` locally — `asciinema` is only required to RE-RECORD the cast (a one-shot maintenance task). Running the demo prints the SSE event stream regardless of whether asciinema is installed.

### docs/agent-architecture.md Planner / Executor section (SC2)

- **D-09:** Insert a new section `## Planner / Executor Model` BEFORE the existing `## Authoring Tools` (Phase 17, currently at line 7). Each section now has its runnable code snippet — closing ROADMAP SC2.

- **D-10:** Section content (~120-150 lines target):
  - 2-paragraph concept overview: Planner is a stateless function `(messages, tools) → ToolPlan`; Executor consumes a `ToolPlan` and dispatches each `parallel_group` via `asyncio.as_completed` while preserving v1.3 `BaseException` isolation (Phase 16 D-01); Synthesizer is the LLM's terminal `call_agentic_turn` after results return.
  - ASCII flow diagram: Request → AgentQueryPipeline → Planner.plan_from_messages → Executor.execute_plan_streaming → Synthesizer (call_agentic_turn) → Response. One short sentence per arrow.
  - `ToolPlan` / `ToolCall` Pydantic V2 signatures verbatim from `utils/models.py` (frozen models, ConfigDict).
  - `Planner.plan_from_messages` signature + `Executor.execute_plan_streaming` signature.
  - **One** runnable snippet: instantiate `AgentQueryPipeline`, call `run_streaming(req)`, iterate the events, log each — copy-pasteable into a Python script.
  - Cross-references: `## Authoring Tools` (how to add a tool the planner can pick), `## Event Schema Reference` (the wire format the executor emits).

- **D-11:** Snippet uses real shipped types (`AgentQueryPipeline`, `GenerationRequest`, `AgentEvent`). NOT pseudocode. NOT a fixture stub. The reader can adapt it to their own auth dependency. ~25-30 lines.
