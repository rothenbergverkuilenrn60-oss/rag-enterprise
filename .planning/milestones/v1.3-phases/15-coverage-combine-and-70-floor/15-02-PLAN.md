<objective>
Backfill unit tests for services/ modules below the 70% combined coverage floor (TEST-06 AC#4) so the CI gate wired by Wave 1 (15-01) exits green. This is a measure-then-plan phase per D-04: Task 1 produces the canonical list of below-70% modules by running combined coverage locally; Tasks 2..N (number determined at execute time) each add 1-2 unit tests per identified module covering primary execution path + at least one error path. The wave ends when `coverage report --fail-under=70` exits 0 against combined data. NO production code (services/, utils/) is modified — Wave 2 is test-only. NO tests for utils/ or controllers/ — TEST-06 AC#4 scope is "service modules" per CONTEXT deferred-ideas.

Purpose: Wave 1 plumbs the floor gate but doesn't enforce 70% until services/ modules are actually covered. v1.2 close shipped at 46% floor; Wave 2 closes the 24-percentage-point gap by backfilling on the modules with the highest gap-impact (lowest current coverage × highest LOC). The plan is structured to support iterative execution: the executor runs Task 1, expands the task list with one task per identified module, and stops when the floor is met. This avoids pre-locking a module list that might be stale by execute time (production code may have been touched in Phase 12/13/14).

Output: N new test files under `tests/unit/` (where N = number of services/ modules identified as below 70%). Each file follows the project's existing unit-test conventions (`@pytest.mark.unit`, asyncio-aware where the module is async, mock fixtures via `__new__` bypass for class-based services per Phase 12 / 13 patterns). One commit per module: `test(15-02): cover services/<module>.py to 70%+`. No production code modified. After the final commit: `coverage combine --keep .coverage.unit .coverage.integration && coverage report --fail-under=70` exits 0.
</objective>
