## Event Schema Reference

The agent runtime emits a structured SSE event stream on `POST /api/v1/agent/v1/run/stream`
(AGENT-04, Phase 18). Each event is serialized as a named SSE frame:

```
event: <event_type>
data: <model_dump_json>

```

Frames are separated by a blank line per the SSE specification. The terminal
event is `synthesizer.final` — there is no `[DONE]` sentinel. Browser
`EventSource` consumers dispatch on the `event:` line via `addEventListener`.

All event payloads are Pydantic V2 frozen models in `utils/models.py`
(`AgentEvent` base + 6 concrete subclasses). Every event carries three common
fields populated by the orchestrator:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trace_id` | string (8-hex) | yes | Per-stream identifier; matches the orchestrator's `trace_id`. |
| `seq` | integer | yes | Monotonic counter per stream; starts at 0; strictly increasing across all event types in a single stream. |
| `ts_ms` | integer | yes | Unix epoch milliseconds at emit time. |

The `event_type` discriminator is a `ClassVar[str]` on each subclass and is
NOT included in the JSON `data:` payload — the SSE `event:` line carries it.

**Redaction policy (Phase 18 D-11):** `tool.span.start.args` is verbatim from
the LLM's tool-call arguments. `tool.span.end.content_preview` truncates the
tool's output to the first 200 characters. `tool.span.error.error_message`
truncates `str(exc)` to the first 200 characters; full tracebacks are logged
server-side only. Multi-tenant isolation is preserved by JWT + Postgres RLS:
each authenticated stream sees only its own tenant's data by construction.

### planner.plan

Emitted once per planner turn, immediately after the planner LLM returns a
non-terminal `ToolPlan`. The terminal plan (no steps; rationale IS the answer)
does NOT emit a separate `planner.plan` event — it goes straight to
`synthesizer.final`.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trace_id` | string | yes | Common — see above. |
| `seq` | integer | yes | Common. |
| `ts_ms` | integer | yes | Common. |
| `plan` | object (`ToolPlan`) | yes | Full planner output: `steps` (list of `ToolCall`), `parallel_groups` (list of int lists), `rationale` (string), `raw_assistant_msg` (provider-shaped dict), `stop_reason` (`text_only` / `tool_use` / `max_tokens` / `error`). |

Example payload:

```json
{
   "trace_id": "abc12345",
   "seq": 0,
   "ts_ms": 1715284800000,
   "plan": {
      "steps": [
         {"id": "call_1", "name": "search_knowledge_base", "arguments": {"query": "leave policy"}}
      ],
      "parallel_groups": [[0]],
      "rationale": "single-hop retrieve",
      "raw_assistant_msg": {"role": "assistant", "content": "..."},
      "stop_reason": "tool_use"
   }
}
```

### tool.span.start

Emitted once per tool dispatch, BEFORE the dispatch coroutine awaits.
Within a parallel group, all `tool.span.start` events for the group fire
before any task in the group resolves.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trace_id` | string | yes | Common. |
| `seq` | integer | yes | Common. |
| `ts_ms` | integer | yes | Common. |
| `span_id` | string (8-hex) | yes | Per-dispatch identifier; matches the corresponding `tool.span.end` or `tool.span.error`. |
| `name` | string | yes | Registered tool name (e.g. `search_knowledge_base`). |
| `args` | object | yes | LLM-provided arguments verbatim. **Not redacted** — see redaction policy above. |

Example payload:

```json
{
   "trace_id": "abc12345",
   "seq": 1,
   "ts_ms": 1715284800100,
   "span_id": "9f3c1e2a",
   "name": "search_knowledge_base",
   "args": {"query": "leave policy"}
}
```

### tool.span.end

Emitted when a tool dispatch resolves to a `ToolResult` (success or controlled
error from the tool). Replaced by `tool.span.error` when the dispatch raises a
`BaseException`.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trace_id` | string | yes | Common. |
| `seq` | integer | yes | Common. |
| `ts_ms` | integer | yes | Common. |
| `span_id` | string (8-hex) | yes | Matches the prior `tool.span.start` for this dispatch. |
| `latency_ms` | integer | yes | Wall-clock milliseconds for this single dispatch (per-tool, not per-group). |
| `chunk_count` | integer | yes | Sourced from `ToolResult.metadata["chunk_count"]` with `len(ToolResult.chunks)` fallback (Phase 17 D-02 metadata convention). |
| `is_error` | boolean | yes | `True` if the tool returned a controlled error result (e.g. RetrieveTool fallback message). |
| `content_preview` | string | yes | First 200 characters of `ToolResult.content`. |

Example payload:

```json
{
   "trace_id": "abc12345",
   "seq": 2,
   "ts_ms": 1715284800650,
   "span_id": "9f3c1e2a",
   "latency_ms": 412,
   "chunk_count": 3,
   "is_error": false,
   "content_preview": "<context>\n[来源1] 员工产假为98天..."
}
```

### tool.span.error

Emitted INSTEAD OF `tool.span.end` when a tool dispatch raises `BaseException`
(including `CancelledError` and `TimeoutError`). The `BaseException` isolation
guarantee from v1.3 D-01 is preserved: siblings in the same parallel group
continue running and emit their own end-or-error events.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trace_id` | string | yes | Common. |
| `seq` | integer | yes | Common. |
| `ts_ms` | integer | yes | Common. |
| `span_id` | string (8-hex) | yes | Matches the prior `tool.span.start` for this dispatch. |
| `latency_ms` | integer | yes | Wall-clock ms before the exception was raised. |
| `error_type` | string | yes | `type(exc).__name__` (e.g. `RuntimeError`, `TimeoutError`). |
| `error_message` | string | yes | First 200 characters of `str(exc)`. Full traceback is logged server-side only. |

Example payload:

```json
{
   "trace_id": "abc12345",
   "seq": 2,
   "ts_ms": 1715284800050,
   "span_id": "9f3c1e2a",
   "latency_ms": 5,
   "error_type": "RuntimeError",
   "error_message": "vector store connection refused"
}
```

### executor.parallel

Emitted once per parallel group at the END of the group, after all of its
`tool.span.end` and `tool.span.error` events have fired. `group_latency_ms` is
the wall-clock duration of the entire group (which is bounded by the slowest
tool — see ROADMAP Phase 18 SC4 latency assertion).

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trace_id` | string | yes | Common. |
| `seq` | integer | yes | Common. |
| `ts_ms` | integer | yes | Common. |
| `fan_out` | integer | yes | Number of tools dispatched in this group. |
| `group_latency_ms` | integer | yes | Wall-clock ms for the entire group (bounded by `max(tool_latency)`, not sum). |

Example payload:

```json
{
   "trace_id": "abc12345",
   "seq": 8,
   "ts_ms": 1715284801000,
   "fan_out": 3,
   "group_latency_ms": 510
}
```

### synthesizer.final

Terminal event. Emitted exactly once per stream, regardless of how the
iteration loop ended (terminal plan, max-iterations cap, or error fallback).
There is no `[DONE]` sentinel — `synthesizer.final` IS the terminal frame.

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `trace_id` | string | yes | Common. |
| `seq` | integer | yes | Common; this is the highest seq in the stream. |
| `ts_ms` | integer | yes | Common. |
| `answer` | string | yes | Composed final answer text. Empty string if all iterations errored. |
| `sources_count` | integer | yes | Total deduplicated chunks accumulated across all tool dispatches in the stream. |

Example payload:

```json
{
   "trace_id": "abc12345",
   "seq": 9,
   "ts_ms": 1715284801200,
   "answer": "员工产假为 98 天 [来源1]...",
   "sources_count": 5
}
```

### Consuming the Stream

Minimal browser EventSource consumer:

```javascript
const es = new EventSource('/api/v1/agent/v1/run/stream');

es.addEventListener('planner.plan', (e) => {
   const evt = JSON.parse(e.data);
   console.log('plan:', evt.plan.parallel_groups, evt.plan.rationale);
});

es.addEventListener('tool.span.start', (e) => {
   const evt = JSON.parse(e.data);
   console.log(`tool start: ${evt.name} (span=${evt.span_id})`, evt.args);
});

es.addEventListener('tool.span.end', (e) => {
   const evt = JSON.parse(e.data);
   console.log(`tool end: ${evt.span_id} ${evt.latency_ms}ms ${evt.chunk_count} chunks`);
});

es.addEventListener('tool.span.error', (e) => {
   const evt = JSON.parse(e.data);
   console.warn(`tool error: ${evt.span_id} ${evt.error_type}: ${evt.error_message}`);
});

es.addEventListener('executor.parallel', (e) => {
   const evt = JSON.parse(e.data);
   console.log(`group done: fan_out=${evt.fan_out} ${evt.group_latency_ms}ms`);
});

es.addEventListener('synthesizer.final', (e) => {
   const evt = JSON.parse(e.data);
   console.log('final:', evt.answer);
   es.close();
});
```

`EventSource` issues a GET by default; this route accepts POST. For POST-based
SSE, use `fetch` with a streaming reader (the `make demo-agent` target in
Phase 19 ships a runnable example).
```
