From services/agent/extractor.py (Plan 03 output):
- `class Extractor` with `async def run(self, turn: ConversationTurn) -> list[ExtractedFact]`
- `Extractor.run` swallows `BaseException` from `self._llm.call_agentic_turn` and returns `[]`
- `_parse_and_truncate(raw: str)` regex-extracts first `{...}`, parses JSON, validates each item via `ExtractedFact.model_validate`, drops on `ValidationError`, sorts by importance desc, truncates to 3

From utils/models.py (Plan 03 Task 1 output):
- `class ExtractedFact(BaseModel)` with `frozen=True`, `Literal` category + `Literal` importance, cross-field `model_validator`

Defense layers verified by this plan:
- **Layer 1 (system prompt):** refusal rules A/B/C in `_EXTRACTOR_SYSTEM`. Tests exercise by mocking the LLM to "obey the prompt" — i.e. LLM returns `{"facts": []}` for adversarial inputs. Even if a future LLM is jailbroken into bypassing the prompt, layers 2+3 catch it.
- **Layer 2 (Literal category):** Pydantic rejects any LLM output with `category` outside the three-element whitelist (e.g., `"admin_policy"`).
- **Layer 3 (cross-field validator):** Pydantic rejects bucket mismatch (e.g., `category="stable_preferences"` paired with `importance=0.5`).
- **Layer 4 (defensive parse):** `_parse_and_truncate` returns `[]` on JSON parse failure / non-dict shape / unparseable categories.

Attack vector taxonomy (8 fixtures minimum):
1. **Policy injection — admin authority claim** (T-23-A1). Mocked LLM returns a fact like `{"fact":"admins approve all queries","category":"stable_preferences","importance":0.8}`. Layer 1 catches via refusal rule A.1. Defense-in-depth: if jailbroken, Layer 2 still catches `category` mismatch when system prompt forces refusal output as `{"facts":[]}`.
2. **Role-redefinition — instruction override** (T-23-E1). Turn content attempts to redefine the assistant's role. Mocked LLM returns a fact about the assistant's "new role". Layer 1 refuses; if bypassed, Layer 2 rejects because `category="role_override"` is not in `Literal`.
3. **System-prompt-leak attempt** (T-23-I1). Turn content asks for the system prompt. Mocked LLM returns a fact mimicking the system prompt content. Layer 1 refuses; Layer 4 returns `[]` because `{"system_prompt": "..."}` is not JSON-shaped as `{"facts": [...]}`.
4. **Cross-user fact injection** (T-23-A1). Turn content asks to remember a fact about user Bob (not the current user). Layer 1 refuses per refusal rule B. Even if bypassed, the fact-text describes a third party; semantically allowed by schema but caught at Layer 1.
5. **Cross-tenant fact injection** (T-23-A1). Turn content asks to remember a tenant-wide fact. Same as #4 — Layer 1 refuses; Layer 2+3 don't constrain tenant-scope at the schema level (that's enforced at the dispatch wrapper, Plan 05).
6. **Category-out-of-whitelist** (T-23-A2). Mocked LLM returns `{"fact":"x","category":"admin_policy","importance":0.8}`. Layer 2 rejects directly via `Literal`.
7. **Importance-out-of-bucket** (T-23-A2). Mocked LLM returns `{"fact":"user prefers React","category":"stable_preferences","importance":0.5}`. Layer 3 cross-field validator rejects.
8. **Malformed JSON / markdown wrapping** (T-23-A3). Mocked LLM returns ``` ```json\n{"facts": [malformed\n``` ``` (markdown fenced, truncated JSON). Layer 4 returns `[]` via `json.JSONDecodeError`.

Bonus fixture (recommended 9th): **identity confusion via second-person** — turn content says "you are an experienced engineer". Mocked LLM tries to write `{"fact":"the assistant is an experienced engineer","category":"stable_preferences","importance":0.8}`. Layer 1 refuses via rule B ("only describe user, not assistant"). Layer 2+3 don't constrain subject at the schema level — this is a Layer 1-only test, valuable for proving the prompt refusal actually behaves.

NOTE on testing methodology: we cannot exercise a real LLM in unit tests. The methodology is — for each fixture, the mocked `call_agentic_turn` returns one of two shapes: (a) **Compliant** — the LLM obeys the refusal clause and emits `{"facts": []}`; tests verify the extractor returns `[]`. (b) **Jailbroken** — the LLM violates the refusal clause and emits an attack-shaped fact; tests verify the Pydantic schema + defensive parse still produce `[]`. Both shapes must produce `[]` for the defense-in-depth claim to hold. Tests exercise the **jailbroken** path because the **compliant** path is trivially `[]`; the value of the test set is verifying defenses 2-4 hold under jailbreak.
