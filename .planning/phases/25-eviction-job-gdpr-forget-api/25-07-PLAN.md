    ---

    **## Eviction — Schedule & Cap**

    Content covering:
    - Default cap: 500 facts per `(user_id, tenant_id)` bucket
    - Env override: `MEMORY_FACTS_CAP_PER_USER=<int>` — set in k8s ConfigMap `rag-config`
    - Cap tuning: Run `--mode=audit` first; observe per-bucket `row_count` and `over_cap_by` in stdout JSON-lines; set cap at 95th-percentile observation + 20% headroom; re-run audit to confirm before enforce
    - Warning: `--mode=enforce` runs immediately on invocation — no code-enforced preflight; operator must run audit mode first (D-3.2)

    **## Audit Mode Workflow**

    Content covering:
    - Step 1: Run `uv run python scripts/evict_long_term_facts.py --mode=audit`
    - Step 2: stdout JSON-lines format: `{"bucket": {"user_id": ..., "tenant_id": ...}, "row_count": ..., "over_cap_by": ..., "would_delete_count": ..., "sweep_run_id": ...}`
    - Step 3: Pipe to jq: `python scripts/evict_long_term_facts.py --mode=audit | jq '. | select(.over_cap_by > 100)'`
    - Step 4: Observe distribution; choose cap; update `MEMORY_FACTS_CAP_PER_USER`
    - Step 5: Re-run audit; confirm `over_cap_by` is acceptable; proceed to enforce

    **## Enforce Mode**

    Content covering:
    - Deletes lowest-importance rows, oldest first among ties (importance ASC, created_at ASC)
    - Chunked at 1000 rows/txn to avoid long lock duration
    - Idempotent: safe to re-run; buckets already at cap produce zero deletes
    - On partial failure: bucket is skipped; next bucket continues; CronJob `restartPolicy: OnFailure` retries the sweep
    - Tie-break example: 3 rows with `importance={0.2, 0.2, 0.8}` and cap=2 — the `0.2` row with the oldest `created_at` is deleted; the `0.8` row and the newer `0.2` row are kept

    **## CronJob YAML**

    Content covering:
    - Embed the §E6 CronJob YAML block verbatim (from 25-RESEARCH.md)
    - Inline YAML comments explaining: `schedule "0 3 * * *"` = daily @ 03:00 UTC; `successfulJobsHistoryLimit: 3` bounds history accumulation; `AUDIT_DB_ENABLED: "true"` writes eviction records to PG `audit_log` (without this, eviction is only in stdout + file log); `restartPolicy: OnFailure` for idempotent retry
    - Note: cap value sourced from `rag-config` ConfigMap — update ConfigMap to change cap without rebuilding image
    - Note: other runtimes (docker-compose, systemd) are operator's responsibility

    **## Forget API**

    Content covering:
    - Purpose: GDPR right-to-be-forgotten for agent-authored facts (long_term_facts ONLY — D-1.2)
    - Scope note: short-term Redis history and user_profile are NOT cleared by this API in v1.6
    - Admin OR self-delete: admin JWT can forget any user; user JWT can only forget their own data
    - Curl example (admin):
      ```
      curl -X DELETE "https://api.example.com/api/v1/memory/forget?user_id=alice" \
        -H "Authorization: Bearer $ADMIN_JWT" \
        -H "X-Confirm-Delete: yes"
      ```
    - Expected response: `{"deleted_row_count": N}` (200 even if N=0 — idempotent)
    - Self-delete curl (non-admin with own JWT): same URL/headers but use user's own JWT where `sub == alice`
    - Error codes: 400 (missing X-Confirm-Delete), 403 (non-admin deleting other user), 404 (empty user_id), 500 (DB failure)
