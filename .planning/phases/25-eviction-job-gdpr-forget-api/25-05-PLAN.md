<objective>
Implement `scripts/evict_long_term_facts.py` — the operator eviction CLI (Wave 2 — depends on Plan 25-01 for `AuditAction.MEMORY_EVICT` + `settings.memory_facts_cap_per_user`).

The script uses the verbatim skeleton from 25-PATTERNS.md Analog 1, which mirrors `scripts/backfill_fact_embeddings.py` with four documented swaps:
1. `--mode=audit|enforce` replaces `--dry-run`
2. `--batch-size` default 1000 (not 100)
3. No `--resume-from-id` cursor (eviction is idempotent-from-start)
4. `DELETE ... WHERE id IN (SELECT id ... ORDER BY importance ASC, created_at ASC LIMIT $N)` replaces `UPDATE ... FROM unnest(...)`

Critical path:
- `LongTermMemory()._get_pool()` for pool (Pitfall 1: register_vector codec)
- `int(status.split()[1])` for row count (Pitfall 2)
- `(asyncpg.PostgresError, asyncpg.InterfaceError)` for the batch-DELETE catch (Pitfall 4 — CLI runs many txns)
- Audit row written AFTER each bucket DELETE, not before (SP-6 / D-2.3)
- `sweep_run_id = uuid.uuid4().hex` propagated to all per-bucket audit rows (D-2.4)
- **T1**: `audit_svc.log()` wrapped in try/except inside `evict_bucket` (both audit-mode and enforce-mode branches). Audit failure must NEVER abort the sweep loop — sweep continues to the next over-cap bucket with a loud structured error log.
- **T8**: `remaining_count` in audit detail dict is computed by re-running `SELECT COUNT(*) FROM long_term_facts WHERE user_id=$1 AND tenant_id=$2` AFTER the chunked DELETE loop. The stale `row_count - total_deleted` arithmetic would lie when save_fact / forget_user runs concurrently in another transaction; re-COUNT keeps the audit row accurate.

Output: one new script + one new unit test file (11 RED→GREEN tests: 10 original + T1 audit-fail-continues).
