## Topic 1 — PaddleOCR for Chinese GB-style PDFs

### 1. Recommendation

**Use `paddleocr>=3.0` with `PP-StructureV3` pipeline (not raw `PP-OCRv5`) on `paddlepaddle==3.0.*` CPU build for Python 3.11; invoke from FastAPI/ARQ via `asyncio.to_thread` with a process-level singleton, and feed it PDF paths directly so PaddleOCR handles its own page rasterization (sidesteps the CMYK PNG bug).**

### 2. Why

- **PaddleOCR 3.x officially supports Python 3.8–3.12** (verified on paddleocr.ai installation page). 3.11 is in-band; no special wheel needed on Linux.
- **PP-StructureV3 accepts `.pdf` paths and `numpy.ndarray` directly** (verified in PP-StructureV3.md): `pipeline.predict(input="./file.pdf")` returns one result per page, with `block_order` for multi-column reading order, `pred_html`+`cell_box_list` for tables, and `rec_formula`+`rec_polys` for equations. This is exactly what GB standards need (section bodies, parameter tables, optical formulas).
- **Page-level rasterization is internal to PP-StructureV3.** It uses PyMuPDF / pdf2image internally to render PDF pages as RGB at a fixed DPI, so the upstream `PyMuPDF.extract_image()` CMYK-to-PNG path (which failed in v1.0 Phase 1) is **not on the OCR critical path**. CMYK only breaks per-image extraction; full-page rasterization renders to RGB regardless of source colorspace.
- **PP-OCRv5 alone is not enough**: it's a text spotter (det+rec) and loses layout/table/multi-column structure. PP-StructureV3 wraps PP-OCRv5 + a layout detector + a table recognizer + a formula recognizer + reading-order recovery — the right granularity for GB documents.

### 3. Implementation sketch

**Install (CPU, baked in Dockerfile):**
```dockerfile
# Pin to known-good versions. paddlepaddle 3.0 is the GA series matching paddleocr 3.x.
RUN pip install --no-cache-dir \
        paddlepaddle==3.0.0 \
        "paddleocr[doc-parser]==3.1.*"

# Bake models into the image to avoid runtime download (~600MB-1.2GB total).
# Pre-warm: run once in build to populate ~/.paddlex/official_models/.
RUN python -c "from paddleocr import PPStructureV3; PPStructureV3()"
```
- Model cache path is `~/.paddlex/official_models/`. Mount or bake — do NOT depend on first-request download in production (warmup is 10–60s and download can fail behind firewalls).
- Use `paddleocr[doc-parser]` extra (lighter than `[all]` which pulls translation models).

**Async wrapper (singleton + thread pool):**
```python
# services/ocr/paddle_ocr.py
import asyncio
from functools import lru_cache
from paddleocr import PPStructureV3

@lru_cache(maxsize=1)
def _pipeline() -> PPStructureV3:
    # PP-StructureV3 is NOT documented as thread-safe; load once per process.
    return PPStructureV3(use_doc_orientation_classify=False,
                         use_doc_unwarping=False,
                         lang="ch")

async def ocr_pdf(path: str) -> list[dict]:
    # PaddleOCR is sync C++/Python; offload to default threadpool.
    # Use a bounded semaphore at call-site to cap concurrency (1-2 per CPU core).
    return await asyncio.to_thread(_run, path)

def _run(path: str) -> list[dict]:
    out = []
    for page_idx, res in enumerate(_pipeline().predict(input=path)):
        out.append({
            "page_number": page_idx + 1,
            "markdown": res.markdown["markdown_texts"],
            "blocks": res.json["res"]["parsing_res_list"],
        })
    return out
```

**Concurrency control in ARQ worker:**
- Use `max_jobs=1` (or `=2` for 8-core hosts) on the OCR-heavy worker — paddlepaddle CPU build will use all cores for matmul; over-subscribing causes thrashing.
- Run OCR jobs in a *dedicated* ARQ queue (`ocr_queue`) separate from ingest — protects fast ingest jobs from being head-of-line blocked.

**Subprocess vs thread pool:**
- **Thread pool is correct** here. Subprocess isolation only matters if you need to recover from segfaults; PaddleOCR 3.x is stable on CPU. Subprocess adds 5–10s startup cost per page (model reload) — fatal for throughput.
- Reserve subprocess strategy for a *retry path* (see pitfalls) — first attempt in-thread, second attempt in subprocess with a 120s hard timeout.

### 4. Pitfalls

- **CMYK is not actually an OCR problem** — only an *image extraction* problem. Decide once: keep extracting embedded images via PyMuPDF for vision-LLM captioning (with the existing CMYK→RGB fix from v1.0 ERR-sweep), AND separately page-rasterize for OCR. Two parallel paths, not one combined.
- **Model warmup latency is 10–60s** on cold start (CPU). Pre-warm in the worker's `on_startup` hook; otherwise the first job in every worker pays the tax.
- **Disk cost: PP-StructureV3 = ~600MB–1.2GB** (layout + table + formula + det + rec models). Bake into image; do NOT rely on Hugging Face download at container start (verified flaky behind enterprise proxies).
- **No documented thread safety.** `lru_cache` singleton + `asyncio.to_thread` + bounded concurrency (semaphore=1) is the safe contract. Do NOT call `predict()` from multiple threads on the same instance.
- **Failure modes — retry vs skip:**
  | Symptom | Action |
  |---|---|
  | `RuntimeError: ... timeout` after >120s on one page | Retry once in subprocess; then skip page, log `ocr_skipped_page` metric. |
  | OOM (paddlepaddle abort) | Do NOT retry in-process (memory may be corrupt). Restart worker. |
  | Empty `rec_texts` for whole page | Skip — likely a blank/illustration-only page. Don't error. |
  | Garbled output (high `rec_scores` but non-CJK chars on a known-Chinese doc) | Detect via simple ratio: `cjk_chars / total < 0.3` → flag for review, retain original text but mark `confidence: low`. |
  | First call after import hangs | Model download in progress — emit log, do not retry. |
- **`use_doc_orientation_classify=True` doubles latency.** Only enable if scans are known-rotated; GB PDFs are usually born-digital or upright scans.
- **GPU build is *not* worth it for this workload** unless you have a dedicated A10/T4 — CPU PP-StructureV3 hits 1–3s/page on modern Xeons, GPU model loading dominates for short docs.

## Topic 2 — Page + Section Metadata Enrichment for Multimodal Chunks

### 2.1 Recommendation

**Store page/section as JSONB metadata only; do NOT prepend "Page 63 — section 3.10:" into the embedded text. Add a query-time *filter extractor* (regex first, LLM fallback) that strips Chinese page/section phrases from the user query and projects them into a pgvector `WHERE metadata @>` filter, combined with `hnsw.iterative_scan = relaxed_order` and an elevated `hnsw.ef_search`.**

### 2.2 Why

- **Prepending structured prefixes dilutes recall.** LlamaIndex's own metadata-extraction docs frame metadata as "context to disambiguate similar chunks" — but the recommended pattern uses `excluded_embed_metadata_keys` to **exclude** noisy fields (page numbers, IDs) from the embedding while keeping them in metadata. Page numbers are high-cardinality, semantically meaningless tokens; embedding them shifts the chunk vector toward an "ID-like" cluster and hurts cosine match against natural-language queries.
- **Section headings are a different story.** A short heading like `第3.10节 灯具发光面要求` IS semantically meaningful and *can* help recall when it shares vocabulary with the query. The clean compromise: prepend **section heading text only** (no page number, no section number prefix like "3.10"), and keep the numeric `section_id` + `page_number` in metadata.
- **pgvector + JSONB filters: post-filter by default, recall risk is real.** Verified on pgvector README: "With approximate indexes, filtering is applied after the index is scanned… If a condition matches 10% of rows, with HNSW and the default `hnsw.ef_search` of 40, only 4 rows will match on average." For `page_number=63` (single page out of hundreds), default `ef_search` will return near-zero results.
- **Iterative scan (pgvector ≥0.8.0) fixes this.** `SET hnsw.iterative_scan = relaxed_order` keeps walking the HNSW graph until the filter yields enough matches. Combine with a B-tree expression index on `(metadata->>'page_number')` for fast filter evaluation.
- **Query-side filter projection beats query-side embedding tricks.** A user typing `第63页灯具的发光面` is asking *two* things: (a) restrict to page 63, (b) find "luminaire light-emitting surface" semantically. Treating it as one embedding query forces the vector to encode "63" — which the corpus chunks won't have if they're embedded cleanly. Splitting `→ {filter: page=63, semantic: "灯具的发光面"}` is the LangChain `SelfQueryRetriever` pattern and is industry standard.

### 2.3 Implementation sketch

**Schema (pgvector):**
```sql
ALTER TABLE chunks ADD COLUMN IF NOT EXISTS metadata JSONB NOT NULL DEFAULT '{}';
CREATE INDEX IF NOT EXISTS chunks_page_idx
    ON chunks ((metadata->>'page_number'));
CREATE INDEX IF NOT EXISTS chunks_section_idx
    ON chunks ((metadata->>'section_id'));
-- HNSW already exists on embedding column.
```

**Per-chunk metadata at ingest:**
```python
chunk.metadata = {
    "page_number": 63,           # int, used for exact filter
    "section_id": "3.10",        # string, used for exact filter
    "section_title": "灯具发光面要求",  # used for embedding context (see below)
    "doc_id": "GB4785-2019",
    "modality": "text" | "ocr" | "image_caption",
}
# Embedded text = section_title + "\n\n" + chunk_text   (heading words help recall)
# Embedded text does NOT contain "第63页" or "3.10"     (numbers hurt recall)
```

**Query-time filter extractor (regex first, LLM fallback):**
```python
# services/nlu/filter_extractor.py
import re

PAGE_RE = re.compile(r"第\s*(\d+)\s*页|page\s*(\d+)|p\.?\s*(\d+)", re.IGNORECASE)
SECTION_RE = re.compile(r"第?\s*(\d+(?:\.\d+){0,3})\s*[节条章]")

def extract_filters(query: str) -> tuple[dict, str]:
    """Return (metadata_filter, residual_semantic_query)."""
    flt: dict = {}
    residual = query

    if m := PAGE_RE.search(query):
        page = next(g for g in m.groups() if g)
        flt["page_number"] = int(page)
        residual = PAGE_RE.sub("", residual)

    if m := SECTION_RE.search(query):
        flt["section_id"] = m.group(1)
        residual = SECTION_RE.sub("", residual)

    residual = re.sub(r"\s+", " ", residual).strip()
    # If residual is empty (pure metadata query), keep original for vector tie-break.
    if not residual:
        residual = query
    return flt, residual

# Optional LLM fallback: only invoke if regex misses AND query contains digits + CJK.
# Avoids per-query LLM cost for the 95% case.
```

**SQL combining filter + vector:**
```sql
SET LOCAL hnsw.iterative_scan = 'relaxed_order';
SET LOCAL hnsw.ef_search = 200;   -- raise from default 40 when filtering

SELECT id, content, metadata, embedding <=> $1 AS distance
FROM chunks
WHERE ($2::int IS NULL OR (metadata->>'page_number')::int = $2)
  AND ($3::text IS NULL OR metadata->>'section_id' = $3)
ORDER BY embedding <=> $1
LIMIT 20;
```
- `relaxed_order` is acceptable for top-k retrieval; switch to `strict_order` only if you need monotonic distance guarantees (we don't — reranker handles final ordering).
- `ef_search=200` is a reasonable starting point when one filter is active; tune via eval set.

**Industry-pattern reference:**
- LangChain `SelfQueryRetriever` does exactly this with `AttributeInfo` schema declaring filterable fields; it uses an LLM to generate the filter. For our Chinese-heavy, narrow query patterns, regex-first is faster and 100% deterministic; reserve LLM for the long tail.
- LlamaIndex `MetadataFilters` is the storage-side equivalent; binding a `FilterCondition.AND` of `MetadataFilter(key="page_number", value=63)` produces the same SQL semantics.

### 2.4 Pitfalls

- **Do not add `page_number` to embedded text "just in case".** Verified anti-pattern via LlamaIndex `excluded_embed_metadata_keys` guidance — high-cardinality numerics dilute embeddings. Even if it seems to "help" on one query, it hurts the other 99%.
- **Section headings >40 tokens hurt too.** If GB section titles are long (some are full sentences), truncate to first 30 chars or just the heading-line, not the entire sub-section header tree.
- **Forgetting `hnsw.iterative_scan` is the silent killer.** Without it, a `page_number=63` filter on a 50k-chunk corpus may return 0 results despite many matching chunks existing. Symptom: "the system can't find page 63 even though I know it's there." Always set per-session or in connection pool init.
- **Pure metadata queries** (e.g., "give me page 63") have no semantic signal. Detect `len(residual_query) < 3` and route to a metadata-only SQL path that skips vector search entirely — `SELECT * FROM chunks WHERE metadata->>'page_number'='63' ORDER BY chunk_index`.
- **Image caption chunks need the same metadata.** When the vision-LLM produces a caption for an embedded image, propagate the host page's `page_number` and `section_id` into the caption chunk's metadata. Without this, image retrieval by page is impossible — exactly the v1.0 bug being fixed.
- **Section IDs in GB docs use multiple formats** (`3.10`, `3.10.2`, `附录A.1`, `表5`). The regex above handles `\d+\.\d+` but misses appendices and tables. Add patterns iteratively from real-query logs; do not over-engineer upfront.
- **JSONB B-tree expression indexes only help equality/range.** `metadata @> '{...}'` containment uses GIN, not B-tree. Pick one access pattern: if you'll always filter by exact `page_number`, use B-tree on `(metadata->>'page_number')`. If you need flexible multi-key containment, use GIN on `metadata`. Don't create both.
- **`ef_search=200` raises latency ~3-5x.** Acceptable for v1.1 (single-tenant, low QPS). Revisit when scaling.

## Open Questions (planner's call)

1. **OCR worker placement** — same ARQ pool as ingest (with separate queue) vs separate container? Adds ops complexity but isolates 600MB-1.2GB memory footprint and CPU saturation. **Lean toward separate container.**
2. **PP-StructureV3 vs Marker / Surya / MinerU** — PaddleOCR is the strongest Chinese choice, but MinerU (also Chinese) ships layout+OCR+formula in one package and may be faster. Out of scope for this fetch budget; flag for a follow-up comparison if PaddleOCR latency proves unacceptable on real GB PDFs.
3. **Section ID extraction at ingest time** — do we trust PP-StructureV3's `block_order` + heading detection, or run a separate heading regex over the markdown output (`^第?\d+(\.\d+)*\s+...`)? The latter is more deterministic for GB-formatted docs. **Recommend: regex over PP-StructureV3 markdown, fall back to V3's heading blocks.**
4. **LLM fallback for filter extraction** — worth the latency? Suggest deferring until regex coverage is measured against real query logs (target: regex catches >90%, then no LLM needed).
5. **`hnsw.ef_search` tuning** — 200 is a guess. Needs an eval set with known page/section ground truth; that's a Phase 2 (eval) deliverable.
6. **Versions are LOW confidence.** PaddleOCR 3.x and paddlepaddle 3.0.0 are the GA series as of early 2026 per the install page, but exact compatible patch pins (e.g., paddlepaddle 3.0.0 vs 3.0.1) need a `pip install --dry-run` check on the actual build host before locking the Dockerfile.
